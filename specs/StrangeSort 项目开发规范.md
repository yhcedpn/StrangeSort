# StrangeSort 项目开发规范

## 整体架构设计目标

1. 实现高内聚，低耦合的功能架构。
2. 遵守开闭原则，一个类应该对扩展开放，对修改封闭。
3. 遵守单一职责原则，一个类应该只有一个发生变化的原因，即一个类只负责一项职责。禁止引入功能繁多的超级类。
4. 遵守里氏替换原则，派生类的对象能够替换其基类的对象被使用，且不破坏程序的正确性。
5. 遵守接口隔离原则，一个类不应该使用它不需要的接口。使用多个专门的接口比使用一个单一的总接口更好，降低依赖耦合。
6. 遵守依赖倒置原则，高层模块不应该直接依赖底层模块，二者都应该依赖其抽象。面向接口编程，不面向具体某个类的实现编程。
7. 遵守迪米特法则，一个对象应当对其他对象有最少的了解。即：一个类尽量降低其成员的访问权限，避免低层耦合。
8. 遵守合成复用原则，尽量使用对象组合或聚合，而不是继承关系来达到软件模块复用的目的。 

## 公开行为契约

1. `README.md` 是项目公开算法行为的唯一契约来源。算法说明、输入输出、异常、随机性、是否原地修改、是否返回原引用或新实例等对外可观察行为，均以 `README.md` 为准。
2. 本规范文件负责约束“如何修改算法实现”，不重复完整抄写各算法定义，避免出现多份行为说明相互漂移。
3. 当 `README.md` 与实现或测试不一致时，必须先明确当前改动属于“修复实现偏离契约”，还是“有意修改契约”。禁止在未说明的情况下静默选择其一。

## 算法行为冻结要求

1. 除非用户明确要求或批准行为变更，否则修改算法实现时，禁止改变任何既有公开可观察行为。
2. 公开可观察行为至少包括以下内容：
   - 输出内容与输出长度规则。
   - 是否保持输入元素的相对顺序。
   - 是否允许重排元素。
   - 是否会删除元素、改写元素或仅调整顺序。
   - 数组重载是否修改输入数组，是否返回原数组引用，还是返回新数组。
   - `List<T>` 重载是否原地修改，以及异常时是否可能部分修改。
   - `comparer == null` 时是否使用 `Comparer<T>.Default`。
   - `random` 未显式传入时是否使用 `Random.Shared`。
   - 随机算法在固定随机种子下是否可复现，以及数组重载和列表重载在相同种子下是否应表现一致。
   - 异常类型、异常触发条件，以及异常发生后输入是否保持不变。
3. 任何仅以“更合理”“更像正常排序”“性能更好”“代码更简洁”为理由的改动，都不能成为改变算法语义的正当依据。

## 各算法不可漂移红线

1. `StalinSort`：输出必须始终是输入的子序列，且保留元素的相对顺序不变。禁止引入交换、重排或额外排序。
2. `ThanosSort`：输出必须始终是输入的有序子序列。禁止把该算法改成直接排序、非随机裁剪，或改变固定随机种子下的可复现性。
3. `EpsteinSort`：过滤规则固定为数值区间 `[0, 18)`；过滤阶段不得受比较器影响。禁止在未修改契约的情况下扩大、缩小或替换该过滤规则。
4. `BrezhnevSort`：输出只能由输入长度决定，不得读取输入元素值来影响结果。禁止把它改成基于原值排序、筛选或重排的算法。
5. `TrumpSort`：行为固定为执行一次随机重排。禁止把它改成多轮随机处理、带筛选的处理，或任何保证结果有序的实现。

## 算法修改流程

1. 修改任一算法前，必须先明确这次改动属于以下哪一类：
   - 修复实现缺陷，但不改变公开契约。
   - 修改公开契约。
2. 若属于“修复实现缺陷”，提交内容必须能说明哪些测试证明契约未发生漂移。
3. 若属于“修改公开契约”，则必须在同一次改动中同步更新以下内容：
   - `README.md` 中对应算法说明。
   - 本目录下受影响的规范条款。
   - 对应测试用例。
4. 禁止只修改实现而不更新契约说明；也禁止只修改文档或测试而保留与之冲突的实现。

## 实现约束

1. 优先保持现有公开 API 形状稳定。未经明确要求，不要新增、删除、重命名或重排公开重载。
2. 未经明确要求，不要改变泛型约束、默认参数语义、枚举合法取值范围及其异常行为。
3. 未经明确要求，不要把当前“返回原引用”的情形改为“总是新分配”，也不要反向修改。
4. 未经明确要求，不要把当前“异常时可能部分修改输入”的行为改成回滚语义，也不要反向修改。
5. 如果为了修复缺陷必须触及上述行为，必须先把该改动按“契约变更”处理，而不是伪装成普通重构。

## 禁止事项

1. 禁止为了让算法“更符合常见排序直觉”而修改奇异算法的核心语义。
2. 禁止为了通过测试而在实现中硬编码特定输入、特定长度、特定随机种子或特定比较器结果。
3. 禁止在没有公开说明的情况下，静默改变边界条件行为，例如空输入、单元素输入、重复值、不可比较类型、非法枚举值、溢出和随机源为空等场景。
4. 禁止在未更新契约和测试前，合并任何会改变公开行为的算法修改。
