# StrangeSortTest 项目开发规范

1. 在编写测试代码时，严禁为了让测试用例通过而引入违反开发目标和开发计划的补丁，例如硬编码一个值让测试用例通过。

## 契约测试原则

1. `StrangeSortTest` 的首要职责是防止算法公开行为发生漂移，而不只是验证“代码当前怎么写就怎么测”。
2. 对外公开行为以 `README.md` 为准。测试应优先覆盖 `README.md` 中已声明的行为、边界条件、异常和示例。
3. 当实现修改但公开契约不变时，必须补充或调整测试来证明契约仍然成立。
4. 当公开契约有意变更时，必须在同一次改动中同步更新 `README.md` 与测试；禁止只改实现不改测试预期，也禁止只改测试去迎合实现。

## 算法修改后的最低测试要求

1. 任何涉及算法实现的改动，至少应覆盖对应算法的数组重载和 `List<T>` 重载。
2. 任何涉及算法实现的改动，至少应覆盖以下一类或多类测试，具体以该算法公开契约为准：
   - `README.md` 中已出现的调用示例。
   - 空输入、单元素输入、重复元素、已满足目标条件的输入。
   - 是否保持相对顺序、是否保持元素多重集、是否只输出子序列、是否改写元素值等不变量。
   - 数组重载是否保持输入不变，是否返回原引用或新数组。
   - `List<T>` 重载是否原地修改，以及异常时可能出现的部分修改行为。
   - `comparer == null`、`random == null`、非法枚举值、不可比较类型、溢出等异常路径。
   - 固定随机种子下的可复现性，以及数组重载和列表重载在相同随机种子下的一致性。

## 各算法建议重点覆盖的不变量

1. `StalinSort`：输出是输入子序列，且相对顺序保持不变。
2. `ThanosSort`：输出是输入的有序子序列；固定种子时结果可复现。
3. `EpsteinSort`：结果仅包含 `[0, 18)` 内的值；过滤不受比较器影响；结果最终按比较器排序。
4. `BrezhnevSort`：结果只由输入长度决定；输入原值不会影响输出；长度超出目标类型表示范围时按契约抛异常。
5. `TrumpSort`：结果与输入长度一致，元素多重集一致；固定种子时结果可复现；数组重载和列表重载在相同种子下内容一致。

## 测试编写约束

1. 禁止编写依赖实现细节但未体现公开契约价值的脆弱测试，例如断言内部局部变量、私有辅助流程或偶然的循环结构。
2. 随机相关测试必须显式控制随机种子，禁止依赖 `Random.Shared` 的即时输出作为断言基础。
3. 涉及异常的测试，除断言异常类型外，还应验证契约中声明的输入是否被修改。
4. 新增或修复 bug 测试时，应优先用最小输入构造能稳定复现问题的用例，避免把多个行为混在一个测试里。
5. 如果发现现有测试没有覆盖 `README.md` 中已写明的行为，应优先补齐缺失覆盖，而不是默认实现当前行为就是正确契约。
