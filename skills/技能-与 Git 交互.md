# 技能：与 Git 交互

## 注意事项

1. 与 Git 交互须使用 git 命令行工具。
2. 除非用户主动提出，否则不要提出将变更提交到 Git 的请求，或者提出附加 Git 标签。
3. 当用户主动提出将变更提交到 Git 的请求（或者要求你制定提交计划）后，检查现有的未提交变更，给出你的意见，包括应当分几个 Git 提交，以及提交信息具体怎么写。
4. 当用户主动提出附加 Git 标签到某个提交（或者要求你制定附加标签的计划）后，检查该提交带来的变更，给出你的意见，以及该标签的附注具体怎么写。

### 涉及到远端仓库时

1. 如果用户希望你将本地变更推送到远端仓库，也将 Git Tag 一并推送。

## Git 工作指导和规范

采用约定式提交规范，以创建清晰的 Git 提交历史，并与语义化版本规则相结合。

### 对 Git 提交的说明和要求

#### 结构

<类型>: <描述>

[可选 正文]

[可选 脚注]

#### 结构化元素

提交说明包含了下面的结构化元素，以表明其意图：

1. **fix** 类型，表示在代码库中修复了一个 bug 或问题，和语义化版本规则中的 PATCH 版本相对应，会导致 PATCH 版本变更。只对测试代码或注释的变更不可以用此类型。
2. **feat** 类型，表示在代码库中新增了一个功能，和语义化版本规则中的 MINOR 版本相对应，会导致 MINOR 版本变更。
3. **build** 类型，用于修改项目构建系统，例如修改依赖库，外部接口，升级运行时版本，修改构建流程和配置等。
4. **chore** 类型，用于对非功能性文件的修改，例如工具配置等。
5. **docs** 类型，用于修改文档，例如修改 README 文件或 API 文档，以及 AGENTS.md 等。对注释，计划文件，规范文件的变更用此类型
6. **style** 类型，用于修改代码的样式，例如调整缩进、空格、空行等。
7. **refactor** 类型，用于功能逻辑不发生改变的代码重构，例如修改代码结构、变量名、函数或方法名，对代码进行性能优化等。如果涉及到测试代码的重构，也用此提交类型。
8. **test** 类型，用于修改测试用例，例如添加、删除、修改代码的测试用例等。

#### 破坏性变更

如果某个提交发生了破坏性变更，则应使用 <类型>!: <描述> 这一结构，并且在脚注中引入 BREAKING CHANGE: <描述> 写出具体的破坏性变更内容。
这和语义化版本规则中的 MAJOR 版本相对应，会导致 MAJOR 版本变更。
破坏性变更可以是任意类型的提交的一部分。

#### 各字段书写要求

描述，指的是对代码变更的简短总结，不可过长。该规则也适用于对 BREAKING CHANGE 的描述。
正文，为代码变更提供额外的上下文信息。
脚注，给提交补充结构化信息。

脚注包含的结构化信息可以有：

1. 说明破坏性变更的 BREAKING CHANGE: <描述> 结构化信息。
2. 关联 Issue 的 Refs: <Issue编号> 结构化信息。

### 对 Git 标签的说明和要求

1. 如果用户要求将标签附加到某个包含了 BREAKING CHANGE 的提交，将其提交信息里的 BREAKING CHANGE 信息包含到标签的附注中。如果没有包含破坏性变更，则不添加附注。
2. 在标签的附注中不要又写一遍版本号。
