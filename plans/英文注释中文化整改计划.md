# 英文注释中文化整改计划

## 已确认约束与口径

1. 仓库约定要求行间注释和 XML 注释使用简体中文。
2. 本次整改只改注释文本，不改公共 API、实现逻辑、测试逻辑和文件布局。
3. 保留技术标识和算法名的英文写法，包括但不限于类型名、成员名、XML 标签、`<see cref="..." />`、`<paramref name="..." />`、`<typeparamref name="..." />`、`Random.Shared`、`Fisher-Yates`。
4. 只将自然语言说明改为简体中文，不将代码片段、表达式和框架约定词强行本地化。
5. 翻译应保持原意等价，不得弱化异常语义、可变性约束、随机性约束或行为边界。

## 已知整改范围

1. `StrangeSort/StrangeSorter.cs`
   - 类型级 XML 注释。
   - 数组重载的 XML 注释。
   - 列表重载的 XML 注释。
   - `MarkRemovedIndexes` 上方的英文行间注释。
2. `StrangeSort/RemovalCountStrategy.cs`
   - 枚举及枚举成员的 XML 注释。
3. `StrangeSortTest/StrangeSorterTests.cs`
   - 当前检查未发现需要整改的英文行间注释或 XML 注释，本次默认不修改。

## 翻译原则

1. 同一概念统一译法，避免同义混用。
2. 优先使用贴近代码语义的简洁表述，避免扩写成说明文。
3. 保留 XML 文档结构，不新增无关标签，不重排标签顺序。
4. `param`、`returns`、`exception` 的语义必须与方法签名和当前行为保持严格一致。
5. 对高风险语句逐句翻译，尤其是以下内容：
   - “有序”的判定定义。
   - 数组与列表版本的可变性差异。
   - 默认比较器失效时的异常触发时机。
   - 列表版本发生异常后可能已被部分修改且不会回滚。

## 实施步骤

### 步骤 1：统一术语与翻译边界

目标：先固定翻译口径，避免后续在重复重载注释中出现不一致表述。

实施内容：

1. 确认“只翻译说明文字，不翻译技术标识和算法名”的执行边界。
2. 为重复出现的概念统一译法，例如“有序子序列”“比较器”“原地修改”“删除策略”等。
3. 明确以下内容保持原样：XML 标签、代码标识符、BCL 类型名、异常类型名、算法名、代码表达式。

完成标准：

1. 后续所有注释可按同一口径直接落地。
2. 不会因为译法不一致导致公共文档读起来像由不同人撰写。

### 步骤 2：整改 `RemovalCountStrategy.cs`

目标：先在范围最小的文件中完成中文化，验证术语是否自然且准确。

实施内容：

1. 将枚举 `RemovalCountStrategy` 的 `<summary>` 改为简体中文。
2. 将 `FloorHalf`、`CeilingHalf` 的 `<summary>` 改为简体中文。
3. 保留枚举名、成员名和代码公式的英文及代码形式不变。

完成标准：

1. 文件内不再包含英文自然语言 XML 注释。
2. 注释仍可准确说明奇数长度时每轮删除数量的差异。

### 步骤 3：整改 `StrangeSorter.cs` 的公共 XML 注释

目标：在不改变 API 结构的前提下，将公开类型和公开方法文档完整改为简体中文。

实施内容：

1. 将类型级 `<summary>` 和 `<remarks>` 逐句翻译为简体中文。
2. 将数组重载的 `<summary>`、`<remarks>`、`<param>`、`<returns>`、`<exception>` 翻译为简体中文。
3. 将列表重载的 `<summary>`、`<remarks>`、`<param>`、`<exception>` 翻译为简体中文。
4. 保持所有 `<see cref>`、`<paramref>`、`<typeparamref>`、`<c>` 引用和代码表达式不变。
5. 对重复重载中含义相同的句子使用统一中文表述，减少同义差异。

完成标准：

1. `StrangeSorter.cs` 中公开 API 的 XML 注释全部为简体中文说明。
2. 数组和列表重载的行为边界、默认值语义、异常语义未被翻译过程改变。

### 步骤 4：整改英文行间注释并做残留扫描

目标：消除源代码中的英文自然语言残留，确保整改范围完整闭合。

实施内容：

1. 将 `MarkRemovedIndexes` 上方关于 `Fisher-Yates` 的英文行间注释改为简体中文说明。
2. 保留 `Fisher-Yates` 算法名的英文写法。
3. 再次扫描 `.cs` 文件中的 `///` 和 `//` 注释，确认没有遗漏的英文自然语言注释。

完成标准：

1. 既定范围内不再残留英文行间注释或英文 XML 注释。
2. 不将代码标识符中的英文误判为需要整改的自然语言注释。

### 步骤 5：验证与收尾

目标：确认整改只影响文档文本，不引入格式错误或构建问题。

实施内容：

1. 检查 XML 注释标签是否闭合正确，避免生成文档相关警告。
2. 复查重复重载的中文文案是否一致。
3. 运行 `dotnet test "StrangeSort.slnx"`，确认注释整改未引入编译或测试问题。

完成标准：

1. 解决方案测试通过。
2. 注释中文化后未引入新的编译错误、测试失败或明显文档格式问题。

## 非目标

1. 不顺带重写、扩写或精简公共文档含义。
2. 不顺带调整命名、实现细节、异常策略或测试覆盖。
3. 不处理本次检查范围外的文档文件、规范文件或计划文件中的英文内容。

## 交付结果

1. 一个仅包含注释文本整改的最小变更集。
2. 中文注释口径统一、可直接指导后续实现与维护。
