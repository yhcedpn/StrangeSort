# BrezhnevSort RewriteAsOrdinalSequence 实现计划

## 目标

为项目新增一个带戏谑命名但语义明确的“排序”能力。该能力面向由数值类型构成的数组或列表，不根据输入元素值进行比较或重排，而是仅根据输入长度，把结果改写为从 `1` 到 `n` 的序数序列。

本次实现需要与当前代码库风格保持一致：

1. 公开类型采用独立命名空间和独立静态类型，不修改现有 `StalinSort`、`EpsteinSort` 与 `ThanosSort` 的公开契约。
2. 数组重载不修改输入数组，而是返回新的结果数组。
3. 列表重载原地修改输入列表，保持与仓库现有列表重载相近的使用感受。
4. 公开方法命名必须诚实反映其行为，不将该能力伪装成传统的基于比较器的排序。
5. 行为必须可测试，且测试用例能够稳定锁定“完全忽略原值、只按长度生成 `1..n`”这一核心语义。

## 设计定稿

### 类型与命名

1. 在 `StrangeSort/BrezhnevSort/` 目录下新增 `BrezhnevSorter.cs`。
2. 命名空间使用 `StrangeSort.BrezhnevSort`。
3. 公开类型名为 `BrezhnevSorter`。
4. 公开方法名为 `RewriteAsOrdinalSequence`，直接表达“将内容改写为序数序列”的行为。
5. XML 注释中可以保留 `BrezhnevSort` 的命名，但必须明确说明它不是传统排序。

### 类型范围与约束

1. 公开泛型方法统一使用 `where T : INumber<T>`。
2. 第一版目标支持具有数值语义的常见类型，例如整数类型、浮点类型、`decimal` 与 `BigInteger`。
3. 第一版不引入 `dynamic`、反射或运行时判型兜底，以避免破坏 AOT 兼容性和类型安全。
4. 结果值通过 `T.CreateChecked` 生成，以获得明确的溢出行为。

### 公开 API

计划提供以下公开重载：

```csharp
public static T[] RewriteAsOrdinalSequence<T>(T[] values)
    where T : INumber<T>;

public static void RewriteAsOrdinalSequence<T>(List<T> values)
    where T : INumber<T>;
```

本次不提供以下公开能力：

1. 不提供 `IComparer<T>` 或 `Comparison<T>` 重载，因为该算法不比较输入值。
2. 不提供 `IList<T>`、`IEnumerable<T>`、`Span<T>` 或 `Memory<T>` 公共重载。
3. 不返回生成数量、原值快照或其他包装结果。
4. 不提供升序、降序或其他策略枚举，因为第一版行为固定为生成 `1..n`。

### 核心语义

1. 对长度为 `n` 的输入，结果固定为 `[1, 2, ..., n]`。
2. 输入元素原有的值完全不会影响输出结果。
3. 数组重载返回承载结果的新数组，不修改输入数组。
4. 列表重载原地把每个位置改写为对应的序数值。
5. 空输入应直接成功，并得到空结果。
6. 单元素输入应得到 `[1]`。

示例：

1. `[999, -3, 42]` 的结果为 `[1, 2, 3]`。
2. `[7]` 的结果为 `[1]`。
3. `[]` 的结果为 `[]`。

### 异常与输入契约

1. 数组或列表输入为 `null` 时，抛出 `ArgumentNullException`。
2. 当 `values.Length` 或 `values.Count` 超过目标类型 `T` 可表示的范围时，应通过 `T.CreateChecked` 抛出 `OverflowException`。
3. 数组重载在生成过程中抛异常时，不得修改输入数组。
4. 列表重载允许在原地改写过程中因溢出而出现部分修改，不要求回滚；需要在文档注释中明确说明这一点。
5. 该算法不依赖默认比较器，因此不存在“只有在实际比较时才抛异常”的契约。

## 实现步骤

### 步骤 1：新增核心公开类型与 API 骨架

目标：建立 `BrezhnevSorter` 的文件、命名空间、XML 注释和公开重载骨架，使其目录结构、命名和说明文档与现有项目风格一致。

实现要求：

1. 新增 `StrangeSort/BrezhnevSort/BrezhnevSorter.cs`。
2. 为数组和列表分别提供一个公开重载。
3. 为全部泛型方法添加 `where T : INumber<T>` 约束。
4. 在类型和方法注释中明确说明：
5. 该能力不是传统排序。
6. 该能力不会读取输入元素值来决定输出内容。
7. 数组重载不会修改输入数组。
8. 列表重载会原地修改输入列表。
9. 列表重载在改写途中若因溢出抛异常，列表可能已经部分修改且不会回滚。

完成标准：

1. 代码文件可编译。
2. 公开 API 名称与计划一致。
3. 注释完整且使用简体中文。

### 步骤 2：实现数组重载的序数生成逻辑

目标：完成数组版本 `RewriteAsOrdinalSequence` 的实际行为，确保返回值仅由输入长度决定，且输入数组保持不变。

实现要求：

1. 对 `values` 执行空值校验。
2. 对空数组直接返回空数组。
3. 分配与输入长度相同的新数组。
4. 使用线性循环把第 `i` 个位置写为 `T.CreateChecked(i + 1)`。
5. 不读取输入数组中的元素值。
6. 不在原数组上做任何原地修改。

完成标准：

1. 数组输入不会被修改。
2. 返回结果长度与输入一致。
3. 返回结果固定为 `1..n`。

### 步骤 3：实现列表重载的原地改写逻辑

目标：完成列表版本 `RewriteAsOrdinalSequence` 的原地处理，保证结果仅由列表长度决定，并保持实现简洁直接。

实现要求：

1. 对 `values` 执行空值校验。
2. 对空列表直接返回。
3. 使用线性循环把第 `i` 个位置改写为 `T.CreateChecked(i + 1)`。
4. 不读取列表原有元素值来参与任何判断。
5. 不引入额外公开辅助类型。

完成标准：

1. 原地处理后的列表长度保持不变。
2. 列表内容固定为 `1..n`。
3. 实现不依赖比较器或其他多余抽象。

### 步骤 4：补齐测试用例并锁定公共语义

目标：在 `StrangeSortTest/BrezhnevSort/` 下新增测试文件，覆盖输入校验、长度驱动语义、可变性语义和溢出行为，确保后续重构不会破坏这套约定。

建议测试清单：

1. `null` 数组输入抛出 `ArgumentNullException`。
2. `null` 列表输入抛出 `ArgumentNullException`。
3. 空数组返回空数组。
4. 空列表处理后保持为空。
5. 单元素数组得到 `[1]`，且输入数组不变。
6. 单元素列表处理后得到 `[1]`。
7. 任意内容但相同长度的两个数组得到相同结果。
8. 任意内容但相同长度的两个列表处理后得到相同结果。
9. 数组重载不会读取原值语义测试，例如 `[999, -3, 42] -> [1, 2, 3]`。
10. 列表重载不会读取原值语义测试，例如 `[999, -3, 42] -> [1, 2, 3]`。
11. 数组重载在成功路径下不修改输入数组。
12. 列表重载在成功路径下原地改写。
13. `BigInteger` 可以正常生成 `1..n`。
14. `double`、`decimal` 等常见数值类型可以正常生成 `1..n`。
15. `byte` 数组长度超过 `255` 时抛出 `OverflowException`，且输入数组不变。
16. `byte` 列表长度超过 `255` 时抛出 `OverflowException`。

完成标准：

1. 测试命名与现有测试风格一致。
2. 测试不依赖随机性。
