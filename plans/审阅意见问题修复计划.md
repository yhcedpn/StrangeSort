# 审阅意见问题修复计划

## 背景

本计划用于处理一次代码审阅中提出的两项低风险问题：

1. `StrangeSort/StrangeSorter.cs` 中只有 `List<T>` 的四参数重载写明了“执行失败后列表可能已部分修改且不会回滚”的公开 contract，而三个便捷重载虽然转调到同一实现，但 XML 注释未同步说明。
2. `StrangeSortTest/StrangeSorterTests.cs` 已覆盖随机源抛错路径，但尚未覆盖比较器抛错路径下的容器差异 contract。

## 已确认结论

1. 这是文档契约缺口，不是算法实现错误。`List<T>` 的三个便捷重载与四参数重载暴露相同的失败语义，因此它们的公开 XML 注释应保持一致。
2. 当前测试缺少 comparer 失败路径覆盖。
3. 数组版本在 comparer 失败时仍应保持输入数组不变。
4. 列表版本在 comparer 失败时只需要验证异常原样传播，不应把“失败后具体内容”或“是否已经部分修改”固化为测试契约。

## 目标

1. 让所有 `List<T>` 公共重载的 XML 注释完整反映相同的失败语义。
2. 为 comparer 失败路径补足最小测试覆盖，且不扩大既有公开 contract。
3. 通过构建与测试确认此次修复没有引入回归。

## 非目标

1. 不修改 `PruneUntilSorted` 的核心算法。
2. 不新增公共 API。
3. 不把列表失败后的中间状态收紧为新的稳定行为承诺。

## 实施步骤

### 步骤 1：补齐 `List<T>` 便捷重载的 XML 注释

目标：消除公开文档与实际失败语义之间的不一致。

实施内容：

1. 修改 `StrangeSort/StrangeSorter.cs` 中以下三个重载的 XML 注释：
   - `PruneUntilSorted<T>(List<T> values)`
   - `PruneUntilSorted<T>(List<T> values, IComparer<T>? comparer)`
   - `PruneUntilSorted<T>(List<T> values, IComparer<T>? comparer, RemovalCountStrategy removalStrategy)`
2. 为上述三个重载补充与四参数重载一致的说明：若剪枝开始后比较器或随机源抛出异常，列表可能已经被部分修改，且不会回滚。
3. 保持现有注释风格与术语，不引入新的文档抽象方式。

完成标准：

1. 四个 `List<T>` 公共重载都明确写出相同的失败语义。
2. 从任一 `List<T>` 重载的 XML 注释都能直接看出该 API 的破坏性边界。

### 步骤 2：补齐 comparer 失败路径测试

目标：让数组与列表在 comparer 抛错场景下的 contract 都有直接测试覆盖。

实施内容：

1. 在 `StrangeSortTest/StrangeSorterTests.cs` 中新增数组测试：
   - 构造一个会在进入剪枝流程后抛出异常的比较器。
   - 调用带显式 `Random` 的数组重载。
   - 断言原始异常被传播。
   - 断言输入数组内容与调用前快照一致。
2. 在同一测试文件中新增列表测试：
   - 使用同类会延迟抛错的比较器覆盖列表原地裁剪路径。
   - 调用带显式 `Random` 的列表重载。
   - 断言原始异常被传播。
   - 不对失败后的列表内容、长度或是否回滚作断言。
3. 复用现有测试风格与最小辅助类型；如需新增测试替身，保持其作用单一且仅服务于 comparer 失败场景。

完成标准：

1. 数组 comparer 失败测试直接验证“输入不变” contract。
2. 列表 comparer 失败测试直接验证“异常传播” contract，且没有把失败后的中间状态误写成稳定行为。

### 步骤 3：执行验证

目标：确认修复后的代码与测试在解决方案内保持通过。

实施内容：

1. 运行 `dotnet build "StrangeSort.slnx"`。
2. 运行 `dotnet test "StrangeSort.slnx"`。

完成标准：

1. 解决方案构建通过。
2. 测试全部通过。
3. 本次变更仅涉及文档和测试，不引入新的实现分支或行为变化。
